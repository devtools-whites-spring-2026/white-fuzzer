# Процесс внесения изменений

## Открытие Pull Request'а

Перед тем, как открыть pull request, **не забудь**
1. Написать тест на добавляемую функциональность. Если теста нет - объяснить, почему
2. Запутить регрессионные тесты и посмотреть, что ничего не сломалось
3. Запустить `style-checker` (я надеюсь, он у нас появится)
4. Проверить, что коммит содержит тег, к чему он относится и (желательно) референс на соответствующую задачу. Пример хорошего `commit message`:
    ```text
    [arch] some description

    Ref: issue#67
    ```

## Внесение исправлений

Если кто-то из команды оставил комментарии, после внесения соответствующих изменений:

1. Закоммить изменения при помощи `git commit --amend`
2. Если в PR несколько коммитов - `git commit --fixup=<commit-hash>`, где `commit-hash` - коммит, к которому нужно добавить изменения. Затем сделай `git rebase --autosquash HEAD~<n>`, где `<n>` - кол-во коммитов в PR вместе с новым от `fixup`
3. `git push --force[-with-lease] origin <your-branch>`, чтобы diff в PR отображал изменения инкрементально, а не все сразу
4. Если есть конфликты с мастером - `git rebase master`

## Если текущая задача заблокирована задачей другого человека

Я надеюсь, что такого не будет, но если произошло:
1. Человек с блокирующей задачей пушит свою ветку [и открывает PR]
2. Человек с блокируемой задачей отводит ветку от блокирующей задачи
3. Если к моменту завершения блокируемой задачи блокирующая еще не влита в мастер - делаем PR блокируемой задачи в PR с блокирующей задачей (если сделать в мастер - будет жуткий diff, не надо)
4. После влития PR с блокирующей задачей PR с блокируемой автоматически перещелкнется на мастер
